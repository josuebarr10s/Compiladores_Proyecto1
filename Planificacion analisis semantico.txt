\section*{El programa tiene sentido}

En esta fase del proyecto de Compiladores, el objetivo ya no es solo reconocer
tokens ni validar la forma del programa. Ahora se busca comprobar si las
instrucciones tienen coherencia dentro del lenguaje del proyecto.

Nuestro lenguaje trabaja con pedidos, clientes, empleados, productos,
cantidades, montos y cambios de estado. Por eso, el an\'alisis sem\'antico debe
revisar si cada acci\'on tiene sentido dentro de ese flujo.

\section{Descripci\'on de la fase}

Hasta este punto, el sistema ya debe reconocer tokens por medio del an\'alisis
l\'exico y validar la estructura general por medio del an\'alisis sint\'actico.

La siguiente pregunta es si el programa tiene sentido.

En esta etapa se busca detectar problemas como los siguientes:

\begin{itemize}[leftmargin=1.5em]
  \item uso de pedidos no creados
  \item redeclaraci\'on de pedidos
  \item incompatibilidad entre campos y valores
  \item asignaciones que no coinciden con lo registrado
  \item uso inv\'alido de identificadores
\end{itemize}

Tambi\'en se dise\~na la tabla de s\'imbolos y se define c\'omo debe construirse
durante el an\'alisis. Esta tabla no se llena de forma manual.

\section{Definici\'on de las reglas sem\'anticas de nuestro lenguaje}

\subsection*{Qu\'e se valida en el an\'alisis sem\'antico}

En esta fase s\'i se revisa si las instrucciones del programa tienen sentido
dentro del lenguaje. El objetivo es comprobar que lo que ya fue aceptado por el
an\'alisis l\'exico y sint\'actico tambi\'en sea coherente en su uso.

\subsection*{Qu\'e no se valida en el an\'alisis sem\'antico}

En esta fase no se revisan errores de escritura ni errores de forma. Esos
problemas se detectan antes, en el an\'alisis l\'exico o en el an\'alisis
sint\'actico.

\begin{itemize}[leftmargin=1.5em]
  \item errores de s\'imbolos no permitidos
  \item cadenas sin cierre
  \item punto y coma faltante
  \item estructura incorrecta de una instrucci\'on
\end{itemize}

\textbf{Resumen claro:} el an\'alisis sem\'antico revisa coherencia. El
an\'alisis l\'exico revisa s\'imbolos y palabras. El an\'alisis sint\'actico
revisa estructura.

\section{Lista de errores sem\'anticos que detectar\'a el compilador}

En esta parte se definen los errores que el compilador debe reconocer durante
el an\'alisis sem\'antico. Cada error representa una situaci\'on en la que el
programa ya fue aceptado en su forma, pero falla en su coherencia interna.

La idea de esta lista es dejar claro qu\'e problemas va a detectar el sistema,
c\'omo se van a identificar y qu\'e tipo de mensaje debe mostrarse cuando uno de
ellos aparezca.

\textbf{\textcolor{red}{SEM-01}} & Pedido no creado
  & Se intenta trabajar con un pedido que no ha sido registrado. \\
\textbf{\textcolor{red}{SEM-02}} & Pedido repetido
  & Se crea dos veces el mismo identificador de pedido. \\

  & Se usa un nombre que no corresponde al contexto donde aparece. \\

\section{Flujo l\'ogico del an\'alisis sem\'antico}

El an\'alisis sem\'antico debe seguir un orden claro y constante. El flujo
propuesto para este proyecto es el siguiente:

\begin{enumerate}[leftmargin=1.5em]
  \item Recibir el resultado del an\'alisis sint\'actico.
  \item Abrir una tabla de s\'imbolos vac\'ia.
  \item Leer las instrucciones en el mismo orden en que aparecen.
  \item Si aparece \mv{Crear Pedido}, verificar que el pedido no exista y luego
        registrarlo.
  \item Si aparece \mv{Empleado}, registrar el nombre para poder usarlo m\'as
        adelante.
  \item Si aparece un campo num\'erico, revisar que el valor sea compatible.
  \item Si aparece \mv{Asignar Empleado}, comprobar que el pedido exista y que
        el empleado ya est\'e registrado.
  \item Si aparece un cambio de estado, comprobar que el pedido exista y que el
        avance tenga sentido.
  \item Si se detecta un error, detener el an\'alisis en ese momento.
  \item Si el recorrido termina sin errores, aprobar el programa.
\end{enumerate}

\begin{lstlisting}[style=code]
INICIO
  |
  v
Recibir arbol sintactico
  |
  v
Abrir tabla de simbolos vacia
  |
  v
Leer la siguiente instruccion
  |
  +-- Crear Pedido      -> validar y registrar
  +-- Empleado          -> registrar
  +-- Campo numerico    -> validar valor
  +-- Asignar Empleado  -> validar pedido y empleado
  +-- Cambio de estado  -> validar pedido y secuencia
  |
  v
Hay error
  |
  +-- Si -> detener analisis
  +-- No -> continuar
  |
  v
FIN SIN ERRORES
\end{lstlisting}

\section{Dise\~no de la tabla de s\'imbolos}

La tabla de s\'imbolos debe guardar solo la informaci\'on que luego ser\'a usada
para validar decisiones del programa. No es una copia completa del archivo, sino
un registro interno de lo que el compilador necesita recordar.

\section{Explicaci\'on de c\'omo se construye la tabla de s\'imbolos}

\subsection*{Cu\'ando se agrega un identificador}

Se agrega cuando el compilador encuentra un dato que luego podr\'a ser usado en
otra parte del programa. En este lenguaje, eso ocurre de forma clara al crear
un pedido o al registrar un empleado.

\subsection*{Cu\'ando se actualiza}

Se actualiza cuando el pedido cambia de estado. Por ejemplo, un pedido puede
pasar de creado a confirmado, luego a procesando y despu\'es a entregado.

\subsection*{Cu\'ando se valida}

Se valida antes de ejecutar cualquier acci\'on importante. Esto ocurre cuando se
quiere cambiar el estado de un pedido, asignar un empleado o aceptar un valor en
un campo num\'erico.

\subsection*{Cu\'ando se detiene el an\'alisis}

El an\'alisis se detiene al encontrar el primer error sem\'antico. Esto permite
mostrar con claridad el problema exacto y el momento en que aparece.

\section{Construcci\'on con error sem\'antico}

\subsection*{C\'odigo de entrada}

\begin{lstlisting}[style=code]
Crear Pedido: S010;
Asignar Empleado S010: Roberto;
\end{lstlisting}

\subsection*{Momento en que ocurre el error}

La primera instrucci\'on crea correctamente el pedido \mv{S010}. El problema
aparece en la segunda instrucci\'on, cuando se intenta asignar al empleado
\mv{Roberto} sin haberlo registrado antes.

\begin{lstlisting}[style=error]
Error Semantico [SEM-04] -- Fila 2
Empleado no registrado: Roberto
No es posible asignar un empleado que no existe en el programa.
Analisis detenido.
\end{lstlisting}

\subsection*{Por qu\'e ocurre}

La acci\'on no tiene sentido dentro del lenguaje porque el compilador no puede
usar un empleado que todav\'ia no forma parte del registro del programa. El
error aparece exactamente al leer la instrucci\'on de asignaci\'on.

\section{Entrega final}

Para la siguiente etapa del proyecto, esta l\'ogica debe quedar implementada en
el compilador con los siguientes elementos:

\begin{itemize}[leftmargin=1.5em]
  \item integraci\'on con el analizador l\'exico
  \item integraci\'on con el analizador sint\'actico
  \item an\'alisis sem\'antico funcional
  \item tabla de s\'imbolos construida autom\'aticamente
  \item detecci\'on de errores sem\'anticos
  \item salida clara del proceso
\end{itemize}
